1. Overview & Logistics
1.1 The Brief
Build TeamSync, a lightweight project & task tracking platform, made up of three connected pieces:
1. A backend API (NestJS + Prisma) that owns the data and business logic.
2. A web app (Next.js) for managers and team members to plan and track work.
3. A mobile companion app (React Native) for team members to view and update their own tasks on the go.
All three pieces should talk to the same backend and the same database. Treat this as a small but real product, not three disconnected exercises.
1.2 Timeline & Submission
* Deadline: 2 calendar days from when you receive this assessment. Need a short extension? Just ask — we'd rather you submit something solid than rush.
* Submission: a GitHub repository (private, with the reviewers below invited as collaborators, or public if you prefer) plus a video walkthrough link.
* Send to: careers@cdazzdev.com with the repo link, video link, and roughly how many hours you spent.
1.3 What We're Looking For
This is an advanced assessment — we expect production-minded decisions, not just a working demo. We care about:
* Whether the system actually works end to end, the way you say it does.
* Code structure, separation of concerns, and how you handle edge cases and errors.
* Whether your design and architecture choices are reasoned, not arbitrary — your README should explain the "why" behind key decisions.
* Security basics: auth, input validation, and not leaking secrets.
* Whether you understand how this would actually run in production on AWS, even though you won't be deploying it for real here.

2. Part A — Backend (NestJS + Prisma)
Build a REST API for TeamSync with the following data model and behavior.
2.1 Data Model
Using Prisma, model at minimum:
* User — id, email, passwordHash, name, role (ADMIN / MANAGER / MEMBER), createdAt
* Project — id, name, description, ownerId, createdAt
* ProjectMember — join table linking users to projects with a per-project role (MANAGER / MEMBER)
* Task — id, projectId, title, description, status (TODO / IN_PROGRESS / DONE), priority (LOW / MEDIUM / HIGH), assigneeId, dueDate, createdAt, updatedAt
* Comment — id, taskId, authorId, body, createdAt
2.2 Required Endpoints
* POST /auth/register, POST /auth/login — issue a short-lived JWT access token and a longer-lived refresh token
* POST /auth/refresh — exchange a valid refresh token for a new access token
* GET /projects — returns only projects the authenticated user belongs to
* POST /projects — Manager/Admin only
* GET /projects/:id/tasks — supports filtering by status, priority, and assignee; pagination (page/limit); sorting by dueDate or priority
* POST /projects/:id/tasks, PATCH /tasks/:id — updates restricted to the task's assignee, the project's manager, or an Admin
* GET /tasks/:id — includes assignee and comment thread
* POST /tasks/:id/comments
2.3 Engineering Requirements
* JWT auth via NestJS Guards, with a role-based access control mechanism (e.g. a custom @Roles() decorator + guard) — not ad-hoc if-checks scattered through controllers
* Request validation via DTOs (class-validator / class-transformer); invalid payloads return 400 with a clear, consistent error shape
* A global exception filter so all error responses share one shape
* Auto-generated API docs via Swagger/OpenAPI, served at /api/docs
* Real Prisma migration files committed (not just schema.prisma) plus a seed script with at least 2 users, 1 project, and 5 tasks
* Jest unit tests covering at minimum the task filtering/pagination logic and the RBAC guard behavior
* A docker-compose.yml that brings up Postgres + the API with a single `docker compose up` — no paid services required
* A .env.example listing every required variable; the real .env must never be committed

3. Part B — Web App (Next.js, Design-to-Code)
This section tests your ability to take a design spec and implement it precisely — the same skill you'd use turning a Figma handoff into production code. The tokens and screens below function as your design spec.
3.1 Design Tokens
* Colors: primary #2563EB, primary-dark #1E40AF, success #16A34A, warning #D97706, danger #DC2626, neutral grayscale 50–900
* Typography: Inter (or system equivalent) — H1 28px/700, H2 22px/600, Body 15px/400, Caption 13px/400
* Spacing scale: 4 / 8 / 12 / 16 / 24 / 32 / 48 px
* Radius: 8px on cards, 6px on buttons and inputs
* Breakpoints: mobile 375px, tablet 768px, desktop 1280px
3.2 Screens
* Login — email/password, inline validation errors, "remember me", link to register
* Dashboard — a project sidebar plus the selected project's tasks (board or filterable list — your choice), with filters for status/priority/assignee
* Task Detail — title, description, status control, priority badge, assignee, due date, and a comment thread with an add-comment form
3.3 Engineering Requirements
* Consumes the live API from Part A — no mocked data in the final build
* Sensible JWT storage strategy (e.g. httpOnly cookie over localStorage) — justify your choice in the README
* Protected routes that redirect unauthenticated users to login
* Loading, empty, and error states on every data-fetching view — not just the happy path
* Fully responsive across all three breakpoints; sidebar collapses to a drawer on mobile
* Client-side form validation (e.g. react-hook-form + zod, or equivalent)
* A clear, justified choice of data-fetching/state approach (React Query, SWR, Context, etc.)

4. Part C — Mobile Companion App (React Native)
Expo is recommended for speed, but bare React Native is acceptable. The mobile app is a focused companion, not a full re-implementation of the web app.
* Login — same auth API; store tokens securely (e.g. expo-secure-store or platform Keychain/Keystore — plain AsyncStorage for tokens is not acceptable)
* Task List — tasks assigned to the logged-in user, with pull-to-refresh and an offline-first cache: the last successful fetch is cached locally and shown with a visible "showing cached data" indicator when the device is offline
* Task Detail — view description, status, and comments; add a comment with an optimistic UI update that rolls back cleanly on failure
* Push notification registration — wire up Expo Notifications to request permission and capture the device push token. You do not need a real push-delivery backend; we're checking that the client-side registration flow is correct

6. Part E — Cloud Architecture (AWS)
No real deployment required — and please don't
* Do not provision real, billable AWS resources for this assessment. We will not reimburse cloud costs, and doing so anyway is treated as not having read these instructions, not as going "above and beyond."
	Instead, write a short architecture document (e.g. ARCHITECTURE.md, 1–2 pages) covering:
* How you'd deploy each of the three components to production on AWS, and which specific services you'd use and why (e.g. ECS Fargate vs. Lambda for the API; RDS vs. Aurora Serverless for Postgres; Amplify/S3+CloudFront vs. an alternative for the Next.js app; how you'd distribute the RN app)
* Your secrets management approach (e.g. Secrets Manager / Parameter Store) — secrets should never live in code or CI logs
* A sketch of a CI/CD pipeline from pull request to production deploy (e.g. GitHub Actions stages)
* One real scaling concern for this system and how you'd address it
Optional, not required and not worth extra risk: Terraform or AWS CDK for the API service + database, validated locally only (terraform validate / cdk synth) — do not run apply or deploy.
